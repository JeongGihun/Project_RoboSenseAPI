## 1. 페이즈 개요

### 1-1. 핵심 전략
```
W1: Claude Code로 RoboSenseAPI 1주 스프린트 (Plan Mode + Execute)
W2~W6: 본인 코드리뷰 + 면접 준비 + 이력서 + 지원
W7~W10: 면접 집중 + 보강 + 오퍼 협상
```

### 1-2. 전략 근거
- **W1 AI 스프린트 채택 이유**: 1년차가 테스트/인증/CI-CD를 10주간 직접 구현해도 "따라 쓴 수준" 이상 되기 어려움. AI 도구 활용 + 코드 이해도가 더 나은 차별화 포인트
- **W2~W6 코드리뷰 비중 큼**: Claude Code가 작성한 코드를 **본인이 완전히 이해하고 설명 가능한 수준**까지. 이게 부족하면 면접에서 AI 작성 티가 남
- **W7~W10 면접 집중**: 면접은 회사 일정에 맞춰야 하므로 후반에 몰림

### 1-3. Claude Code 사용 방식 (Plan Mode + Execute)
- **Plan Mode**: Claude Code가 작업 전 **실행 계획을 먼저 제시** → 사용자가 검토/수정
- **Execute**: 승인된 계획에 따라 실제 코드 수정/생성 실행
- **대안 고려 → 기각**: Subagent 방식은 세팅 1~2일 소요로 W1 일정에 부담

### 1-4. 레포 내 Claude Code 운영 구조

레포 루트에 아래 구조를 둠. Claude Code는 이 구조 기반으로 동작:

```
RoboSenseAPI/
├── CLAUDE.md                         ← 프로젝트 상시 규칙 (TDD, 사이클, Harness)
├── .claude/
│   ├── tasks/
│   │   ├── todo.md                   ← Plan 결과 누적
│   │   └── lessons.md                ← 실수 학습 누적 (Harness)
│   └── context/
│       ├── PHASE1.md                 ← 전체 맥락 (이 파일)
│       └── PHASE1_w1_prompts.md      ← W1 일자별 프롬프트
└── app/ test/ ...
```

각 파일의 역할은 `CLAUDE.md` 참조.

### 1-5. 작업 사이클 (모든 일자 공통, 9단계)

1. **Plan 작성** (Plan Mode)
2. **Task화** (`.claude/tasks/todo.md`에 기록)
3. **테스트 작성** (TDD RED)
4. **구현** (TDD GREEN)
5. **리팩토링** (TDD REFACTOR)
6. **QA** (pytest 전체 실행)
7. **학습 기록** (`lessons.md` 갱신)
8. **커밋** (feature 브랜치)
9. **로컬 검증** (Docker 재빌드 + /health)
10. **integration 머지** (feat/* → integration PR & 머지, CI 통과 확인)

배포: integration → main 머지 시 deploy.yml 자동 트리거. 세부는 `CLAUDE.md` 2번 섹션 참조.

### 1-6. TDD 방침

- **엄격 적용**: 비즈니스 로직, 모델/스키마 검증, 예외 핸들러, 인증/권한
- **예외 허용**: 인프라 설정 (Dockerfile, docker-compose, nginx.conf), CI/CD 파이프라인, 로깅 포맷 자체
- **이유**: 인프라 코드는 환경 의존성 커서 테스트가 역효과. 1년차에게 **"엄격 TDD + 실용적 예외"** 는 면접에서 "TDD 어떻게 적용?" 질문에 구체 답변 가능한 스토리

### 1-7. 브랜치 전략 (W1)

```

  ↑ (토요일 1회 통합 PR)

  ↑
feat/w1-tests         ← D1~D2
feat/w1-api-key       ← D3
feat/w1-logging       ← D4
```

배포는 **W1 종료 후 토요일 저녁** 또는 **일요일**에 `integration → main` PR 머지로 **1회만** 트리거.

### 1-8. Harness (학습 누적)

`.claude/tasks/lessons.md` 에 잘못 판단했던 것들 자동 누적. 매 작업 시작 전 읽음.

기록 트리거:
- 사용자 negative feedback ("아니", "다시 해", "틀렸어" 등) 감지 시 자동 초안 작성
- Plan이 이전 lesson과 충돌하면 경고 후 재설계
- 작업 후 자체 회고 → 학습 있으면 제안

세부는 `CLAUDE.md` 5번 섹션 참조.

> **Claude Code Plan Mode + Execute 방식으로 진행**. 각 작업 시작 전 Plan 검토 → 승인 후 Execute.

### 3-1. 작업 순서 (6일)

| 요일 | 작업 | TDD | 산출물 |
|------|------|------|--------|
| 월 (04.20) | 테스트 코드 기초 (conftest.py, 핵심 엔드포인트) | - (자체가 테스트) | pytest 기본 실행 가능 |

**이 순서의 근거**:
1. **테스트 먼저**: 이후 변경사항이 기존 기능 안 깨는지 검증 가능
2. **API Key 다음**: 인증 없으면 로깅/배포가 무의미
3. **로깅 그 다음**: 배포 전 운영 기반 마련
4. **CI/CD+EC2 마지막 주요 작업**: 모든 게 갖춰진 상태에서 배포 자동화
5. **OpenAPI는 토요일 시작**: 마감 작업이라 W2~W6에 이어가도 됨

### 3-2. 각 작업 상세

#### W1-D1·D2: 테스트 코드
**목표**: pytest 기반 단위/통합 테스트, 커버리지 70%+

**작업 범위**:
- `conftest.py`: pytest + httpx 설치. SQLite 인메모리 DB 오버라이드 fixture. `AsyncClient`로 FastAPI 테스트 클라이언트
- `test/test_sensors.py`: POST/GET 정상/에러 케이스, 필터링, 페이지네이션
- `test/test_robots.py`: 등록/조회/개별 조회, 404 케이스
- `test/test_stats.py`: Redis 캐시 동작, 캐시 무효화 시나리오
- 공통 fixture로 중복 제거

**성공 기준**:
- `pytest -v` 전체 통과
- `pytest --cov` 리포트에서 70%+ 달성
- 에러 응답 포맷 일관성 검증 (404, 422, 500)

**Claude Code 지시 예시**:
> "conftest.py부터 만들어. SQLite 인메모리 기반 test DB fixture + AsyncClient fixture. 각 테스트마다 세션 독립. Plan 먼저 보여줘."

#### W1-D3: API Key 인증
**목표**: JWT 대신 X-API-Key 헤더 기반 간단 인증

**선택 근거** (면접 스토리 자산):
- RoboSenseAPI는 로봇 ↔ 서버 시스템 간 통신. 사람 로그인 아님
- IoT/센서 도메인에서 JWT보다 API Key가 표준에 가까움
- 면접 Q: "왜 JWT 안 썼어요?" → A: "로봇이 로그인할 일이 없고, 센서 시스템은 API Key가 더 자연스럽다고 판단"

**작업 범위**:
- `app/auth.py`: FastAPI `Depends()` 기반 API Key 검증 함수
- `app/models/db_models.py`: `ApiKey` 모델 추가 (key_hash, robot_id, created_at, revoked)
- 발급/폐기 엔드포인트 (관리자용, `POST /admin/api-keys`)
- 기존 엔드포인트에 `Depends(verify_api_key)` 추가
- API Key는 **해시로 저장** (bcrypt 또는 SHA-256 + salt)

**성공 기준**:
- 유효 키 → 200, 무효 키 → 401, 키 없음 → 401
- 인증 테스트 5개+ 추가
- Swagger UI에서 Authorize 버튼으로 테스트 가능

#### W1-D4: 구조화 로깅 + /health + /metrics
**목표**: 운영 관점에서 보여줄 수 있는 기반 마련

**작업 범위**:
- `app/logging.py`: JSON 포맷 로거 (timestamp, level, request_id, message, extra)
- 모든 라우트에 request_id 자동 부여 (미들웨어)
- `GET /health`: DB + Redis 연결 상태 반환
- `GET /metrics`: 기본 메트릭 (요청 수, 응답 시간 p50/p95/p99). Prometheus 포맷 선택사항

**성공 기준**:
- 로그가 JSON 한 줄로 찍힘 (파싱 가능)
- `/health` 응답: `{"status": "ok", "db": "ok", "redis": "ok"}`
- `/metrics` 접근 가능

**주의**: Prometheus 호환까지 가면 복잡해짐. **기본 메트릭 엔드포인트만** 만들고 Prometheus 연동은 후순위

#### W1-D5: CI/CD + EC2 자동 배포 파이프라인 구축
**목표**: main 머지 시 자동 배포되는 파이프라인 준비. **실제 배포는 W1 종료 후 토요일/일요일에 1회 main 머지로 트리거**.

**작업 범위**:
- `.github/workflows/ci.yml`: PR 시 pytest + ruff 자동 실행
- `.github/workflows/deploy.yml`: main 머지 시
  1. Docker 이미지 빌드
  2. GitHub Container Registry에 푸시
  3. EC2에 SSH → `docker-compose pull && docker-compose up -d`
- EC2 설정: docker + docker-compose 설치, 접근 권한, SSH 키 GitHub Secrets에 등록
- **참고**: D5 자체도 feature 브랜치에서 진행. main에 직접 머지 금지

**성공 기준**:
- PR 생성 → feature 브랜치에서 테스트 자동 실행 결과 표시
- 파이프라인 dry-run 또는 스테이징 검증 (가능하면)
- README에 배포 배지 추가 준비
- **실제 main 머지 → EC2 반영은 W1 종료 후** (아래 "W1 종료 후 배포" 참조)

**리스크**:
- EC2 SSH 세팅에서 막힐 수 있음 (보안 그룹, 키 권한 등)
- **대응**: 금요일 오후까지 안 되면 토요일 오전 집중. 그래도 안 되면 CI만 먼저 완성, 배포는 수동 폴백

#### W1-D6: OpenAPI + 커스텀 예외 핸들러
**목표**: Swagger UI 완성도 + 일관된 에러 응답

**W1에 할 것 (토요일)**:
- 각 엔드포인트에 `response_model`, `responses`, `summary`, `description`, `tags` 추가 시작
- 커스텀 예외 클래스 정의 (`SensorTypeInvalidError`, `ApiKeyMissingError` 등)
- 글로벌 예외 핸들러: 에러 응답 포맷 통일
  ```json
  {
    "error_code": "SENSOR_TYPE_INVALID",
    "message": "지원하지 않는 센서 타입입니다",
    "detail": {"sensor_type": "xyz"},
    "timestamp": "2026-04-20T10:30:00Z"
  }
  ```

**W2~W6에 마감**: 모든 엔드포인트에 문서 붙이기, 에러 코드 목록 정리

### 3-4. W1 종료 후 배포 (토요일 저녁 ~ 일요일)

### 4-2. 코드리뷰 방법

**핵심 원칙**: "이 코드를 왜 이렇게 짰는지 **내 말로** 설명 가능해야 한다"

**방법**:
1. **파일 단위로 순회**: `test/` → `app/auth.py` → `app/logging.py` → `.github/workflows/` → 기타
2. 각 파일마다:
   - 전체 구조 파악 (10분)
   - 주요 함수/클래스를 한 줄씩 설명하며 읽기 (30~60분)
   - 모르는 부분 → Claude에게 질문 (별도 대화로)
   - `REVIEW_NOTES.md`에 "왜 이렇게 짰는가" 자기 말로 기록
